Week3 Crawling

URL Frontier Using Mercator Scheme*

Mercator scheme

  1. Prioritizer(freshness) appends URLs to one of k queues
    1. the first means we need crawl it very frequently, the k-th means very infrequently
    2. the assigned queue is decided by
      1. refresh rate sampled from previous crawls
      2. application-specific (e.g. “crawl news sites more often”)
  2. queues are FIFO
  3. queue selector picks a front queue to pull a URL by some strategies
    1. random number: if in [0,0.5) choose 1th queue, if in [0.5, 0.75) choose 2nd queue and so on
    2. [第一次: pick 1th],[第二次: pick 1th 2nd],[第三次: pick 1th 2nd 3rd] and so on
  4. back queues invariants
    1. each is kept non-empty while the crawl is in progress
    2. each only contains URLs from a single host
    3. maintain a table from hosts to back queues(Map<Host name, Back queue>)
  5. Min Heap of Size B, the entry is Map<Host name, the earliest time that the host can be hit again>
    1. one entry for each back queue
  6. Mercator recommendation: three times as many back queues as crawler threads(队列是线程的三倍)

Week3 De-duplicate

Locality Sensitive Hashing*
start from 14 mins

Similar Documents

Three essential techniques:

  1. k-Shingle: convert documents to sets
    1. e.g. k=2;doc=abcab => set{ab,bc,ca}
    2. changing a word affects k shingles
    3. reordering paragraphs only affects 2k shingles that cross paragraph boundaries
    4. compress shingles to 4 bytes by hashing, which is tokens
  2. Minhashing: convert large sets to short signatures
    1. Jaccard Similarity
    2. From sets to boolean matrices
      1. Rows = elements of the universal set (e.g. k shingles)
      2. Columns = sets (e.g. docs)
      3. the matrix is sparse
    3. Signature matrix
      1. Rows = minhash value of k shingles
      2. Columns = sets (e.g. docs)
      3. 以黄色为例，1的时候填好两个位置，2的时候第四个已经占有了，只能填第一个位子，以此类推，到4的时候填好第三个位子
    4. permutation is too slow when encountering 1 billion entries
      1. pick 100 hash functions
      2. for each column c and each hash function h_i keep a slot M(i,c)
      3. M(i,c) become the smallest value of h_i® for which column c has 1 in row r

  3. Locality-sensitive hashing: focus on pairs of signatures likely to be similar

Document Frequency(df)

  1. Rare terms are more informative than frequent terms(stopwords) (e.g. arachnocentric)
    A document containing this word is very likely to be relevant to the query.
    So it should have a high weight
  2. Hence frequent terms are less informative than rare terms (e.g. high, increase, line)
  3. Inverse Document Frequency(idf): idf_t = log_10(N / df_t)（符合一二，pdf里有例子）
  4. idf has no effect on ranking one term queries(at least two terms in queries)

Word Tokenization*

  1. cat and cats: same lemma, but different wordforms
    e.g. my name is my ronan
  2. Type: an element of the vocabulary (4: my name is ronan)
  3. Token: an instance of that type in running text (5: my*2 name is ronan)
  4. tr 'A-Z' 'a-z' < shakes.txt | tr -sc 'a-z' '\n' | sort | uniq -c | sort -n -r | less
    1. 大写变小写->把非字母变换行->排序->统计->根据数量排序
  5. language issues e.g. what’re -> what are; San Francisco(one token or two?); PhD
  6. tokenization in chinese(aka. word segmentation): Standard baseline segmentation algorithm(greedy)
    1. Given a word list of Chinese, and a string
    2. Start a pointer at the beginning of the string
    3. Find the longest word in dictionary that matches the string starting ar pointer
    4. Move the pointer over the word in string
    5. Go to 2
    6. Doesn’t work in English sometimes e.g.
      Thecatinthehat(The cat in the hat);
      Thetabledownthere(The table down there, Theta bled own there)

Word Normalization and Stemming*

Normalization

  1. Normalize(e.g. U.S.A.->USA)
    1. Case folding(ppt)
    2. Lemmatization(ppt): reduce inflections or variant form to base form
  2. Define equivalence classes of terms(e.g. deleting periods(句号) in a term)
  3. Alternative(e.g. Enter: windows -> Search: window, windows)
  4. Morphology
    1. Stems: The core meaning-bearing units
    2. Affixes: Bits and pieces that adhere to stems(often with grammatical functions)
    3. e.g. Stems -> Stem & s
  5. Stemming: Reduce terms to their stems in IR
    1. Porter’s algorithm(ppt) : the most common English stemmer; strip -ing if there is a vowel(e.g. walking->walk,sing->sing)
    2. tr -sc 'A-Za-z' '\n' < shakes.txt | tr 'A-Z' 'a-z' | grep '[aeiou].*ing$' | sort | uniq -c | sort -n -r | less

Week6 Building an Inverted Index

Word Similarity

  1. Synonymy: a binary relation
  2. Similarity(or Distance): a looser metric; a relation between senses(e.g. bank is similar to fund)
  3. Relatedness: car is similar to bicycle; car is related to gasoline, not similar
  4. compute similarity over both words and senses

Two similarity algorithm

  1. Thesaurus-based algorithm
    1. Path based similarity: two senses are similar if they are near each other in the thesaurus hierarchy
      1. pathlen(c1,c2) = 1 + number of edges in the shortest path in the hypernym graph between sense nodes c1 and c2
      2. simpath(c1,c2) = 1 / pathlen(c1,c2)
      3. wordsim(w1,w2) = max sim(c1,c2) where c1 \in senses(w1), c2 \in senses(w2)
      4. e.g. simpath(nickel,coin) = 1/2 simpath(nickel,money) = 1/6
      5. Problem:
        1. nickel to money seems to be closer than nickel to standard, but their simpaths are the same
        2. nodes high in the hierarchy are very abstract
      6. information content similarity metrics
        1. P© = count(w) / N
  2. Distributional algorithm

Word similarity distributional*

  1. Problems with thesaurus-based meaning
    1. We don’t have a thesaurus for every language
    2. Even if we do, many words and phrases are missing
  2. Aka Vector-space models
    1. Two words are similar if they have similar word contexts
    2. Term-document matrix
      Num of Terms in docs
      1. For books, Julius Caesar and Henry V are similar
      2. For words, fool and clown are similar
      3. Use tf-idf instead of raw term counts
    3. Term-context matrix
      Two words are similar if they have similar word contexts
      1. apricot and pineapple are similar, digital and information are similar
      2. Positive Pointwise Mutual Information(PPMI) is common


      3. Weighted PMI
